# Module 4 — data provenance

This module mixes **one real host** (for verifiable answers) with **two synthetic benign peer hosts** (so stacking/LFO has something to count against). Stacking only demonstrates with multiple hosts, and there is no clean, license-clear public multi-host Amcache set — so the peers are generated, not downloaded, and are clearly labelled synthetic.

## Real host — `DESKTOP-SDN1RPT` (the one with the malware)
- `amcache_host-DESKTOP_UnassociatedFileEntries.csv` and the other `amcache_host-DESKTOP_*.csv` (Associated, Program, Device*, Drive*, ShortCuts) — AmcacheParser output.
- `shimcache_host-DESKTOP.csv` — AppCompatCacheParser output (266 entries).
- **Origin:** the same `DESKTOP-SDN1RPT` host from **DFIR Madness "Case 001 — The Stolen Szechuan Sauce"** used in Modules 1-3.
- **Source / license:** https://dfirmadness.com/the-stolen-szechuan-sauce/ — education/defensive use only; respect the author's terms.

## Synthetic benign peers — `WORKSTATION-07`, `WORKSTATION-12`
- `amcache_host-WORKSTATION07_UnassociatedFileEntries.csv`, `shimcache_host-WORKSTATION07.csv`
- `amcache_host-WORKSTATION12_UnassociatedFileEntries.csv`, `shimcache_host-WORKSTATION12.csv`
- **These are SYNTHETIC and contain NO malware.** They are a counting baseline only — never treat them as real evidence.
- **What's in them:** the same common Microsoft System32 binaries as the real host (`svchost.exe`, `csrss.exe`, `winlogon.exe`, `drvinst.exe`, `MsMpEng.exe`, etc.). The shared binaries deliberately reuse the **real host's SHA1 and metadata**, because identical files genuinely share a SHA1 — this is what makes stacking on SHA1 return Count=3 for OS binaries and Count=1 for the malware. Each peer also has a few unique benign apps (Chrome/Outlook/Teams on WS-07; Firefox/7-Zip/VS Code on WS-12) so it looks like a real workstation.
- **License:** generated for this lab; freely reusable. No third-party data.

## How the peers were produced
Generated by a small script that read the real host's Unassociated CSV, copied the Microsoft OS-binary rows (name + SHA1 + size + metadata) into each peer, excluded `coreupdater.exe` and the responder's `FTK Imager.exe`, and appended a handful of invented benign apps. The script is documented in the project history; the output is plain CSV matching the AmcacheParser / AppCompatCacheParser column formats.

## Why this design
- `coreupdater.exe` (and its SHA1 `fd153c66386ca93ec9993d66a84d6f0d129a3a5c`) exists on **one** host → Count = 1 (the LFO outlier).
- Ubiquitous Microsoft binaries exist on all three → Count = 3 (noise).
- A few benign apps are also Count = 1 — the deliberate lesson that **rarity finds candidates, metadata convicts**.

See `get-data.sh` for how to add still more hosts.
